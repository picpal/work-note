# 공유 링크 '대상' 사용자 검색(@) 설계

**작성일:** 2026-06-18
**상태:** 승인됨(설계) → 구현 계획 대기

## 목표

공유 링크 생성 모달의 `대상 사번(콤마 구분)` 자유 입력을, `@`로 사용자를 검색해 **칩(태그)** 으로 선택하는 UX로 교체한다. `@` 입력 시 사용자 목록 드롭다운이 `사번 이름` 형식으로 뜨고, 클릭/Enter로 칩이 추가된다. 공유 grant 계약(`pinEmps`)과 공유 로직은 불변 — **입력 UX만 교체**한다.

## 배경 / 현재 상태

- `ShareModal.tsx`의 `대상 사번(콤마 구분)` input → `pins` 문자열을 콤마 분리해 `body.pinEmps`(사번 배열)로 전송.
- 공유 생성은 `VaultGuard.requireShare(user, id)`로 게이트된 **일반 사용자**도 가능(관리자 한정 아님).
- 사용자 목록 API는 `GET /api/admin/users`(`AdminGuard.requireAdmin`, 전체 `UserRow` 반환 — email·role·status 포함)뿐. **비관리자가 쓸 디렉토리 엔드포인트가 없다.**
- ShareModal은 http(server) 모드 전용(App 컨텍스트 메뉴에서 가드). local 모드는 사용자 개념 없음 → 해당 없음.

## 결정 사항(확정)

1. **대상 표현 = 칩(태그) 방식.** 선택 시 `사번 이름 ×` 칩 추가, Backspace로 마지막 칩 제거. 제출 시 칩들의 emp 배열 → `body.pinEmps`(계약 동일).
2. **디렉토리 접근 = 로그인한 모든 사용자.** 사내 직원 디렉토리 성격. emp+name만 반환, active만. 민감필드(email·role·status·id) 미노출.
3. **디렉토리 선택만.** 드롭다운에 없는 사번을 수동 입력해 칩으로 추가하는 것은 불가 — 모든 칩은 실재 active 사용자임이 보장된다.
4. **검색 = 전체 active 1회 로드 + 클라이언트 필터(접근법 B).** 스펙상 규모(3~4팀, 수십~수백)에서 전체 목록이 가볍고, 디바운스 없이 즉시 필터가 더 단순·빠르다.
5. **활성 링크 목록의 기존 `pinEmps` 표기(raw 사번)는 범위 외** — 이번 변경은 생성 폼 입력 UX만.

## 아키텍처

### 백엔드

신규 비관리자 디렉토리 엔드포인트.

- **컨트롤러:** `com.worknote.auth.UserDirectoryController`(AuthController와 같은 패키지 — 비관리자 user/auth 도메인)
  - `GET /api/users/directory` → `List<DirectoryUser>`
  - 인증 요구만(server 모드 AuthFilter가 세션 사용자 적재). 미인증 → 401(기존 AuthFilter/예외 처리 경로 그대로). 추가 capability 게이트 없음.
  - 별도 감사 로그 없음(읽기 디렉토리 조회 — 기존에 read 류는 감사 안 함).
- **DTO:** `com.worknote.auth.DirectoryUser` — `record DirectoryUser(String emp, String name)`. 직렬화 시 emp·name만.
- **매퍼:** `UserMapper.findActiveDirectory()`
  - `SELECT emp, name FROM app_user WHERE status = 'active' ORDER BY emp` → `List<DirectoryUser>`.
- **불변:** `ShareController`, `ShareLinkService`, `CreateShareRequest.pinEmps`, `VaultGuard.requireShare` — 손대지 않는다.

### 프런트엔드

- **API 클라이언트:** `UserApi.directory()` → `req<DirectoryUser[]>("/users/directory")`. `DirectoryUser = { emp: string; name: string }`.
- **순수 헬퍼(파일: `frontend/src/lib/userDirectory.ts`):**
  - `formatUser(u: DirectoryUser): string` → `` `${u.emp} ${u.name}` ``.
  - `filterDirectory(all, query, excludeEmps, limit)` → query를 trim·소문자화해 emp 또는 name에 부분일치하는 active 사용자 중, `excludeEmps`(이미 선택된 칩) 제외, 앞에서 `limit`개. 빈 query면 제외 후 앞에서 `limit`개.
- **컴포넌트:** `UserPicker`(칩 멀티셀렉트)
  - props: `value: string[]`(선택된 emp), `onChange: (emps: string[]) => void`, `directory: DirectoryUser[] | null`(로딩 전 null), `disabled?: boolean`.
  - 칩 = `사번 이름 ×`(emp→directory에서 name 조회해 표기; 없으면 emp만). 내부 값은 emp.
  - 드롭다운 오픈 규칙: **입력 문자열에 `@`가 있으면 오픈**, 쿼리 = 마지막 `@` 이후 텍스트. `@`가 없으면 닫힘. 빈 쿼리(`@`만) = 상위 N(=8)명.
  - 키보드: ↑/↓ 항목 이동, Enter 선택(칩 추가 후 입력·드롭다운 초기화), Esc 닫기, 입력이 빈 상태에서 Backspace → 마지막 칩 제거.
  - 마우스: 항목 클릭 = 선택. 외부 클릭 = 드롭다운 닫기.
  - 중복 방지: 드롭다운은 이미 선택된 emp 제외(`filterDirectory`의 excludeEmps).
- **ShareModal 변경:**
  - `pins`(string) 상태 → `pinEmps`(string[]) 상태로 교체.
  - 모달 마운트 시 `UserApi.directory()` 1회 로드 → `directory` 상태. 실패 시 `directory=[]` + 인라인 메시지.
  - `대상 사번(콤마)` input 블록 → `UserPicker` 렌더로 교체. 라벨 `대상 (비우면 전 직원)`.
  - `create()`에서 `body.pinEmps = pinEmps`(비었으면 미설정 → 전 직원).

## 데이터 흐름

1. ShareModal 오픈 → `UserApi.directory()` → active `[{emp,name}]` 로드.
2. 사용자가 입력란에 `@김` 입력 → `filterDirectory(directory, "김", 선택된emps, 8)` → 드롭다운.
3. 항목 선택 → `onChange([...value, emp])` → 칩 추가, 입력 초기화.
4. `링크 만들기` → `pinEmps` 배열 → `ShareApi.create(note.id, { …, pinEmps })`(기존 계약).

## 에러 처리

- 디렉토리 로드 실패: `UserPicker`에 빈 directory + 인라인 메시지("사용자 목록을 불러오지 못했습니다"). 디렉토리 선택만이므로 칩 추가 불가, 만료·열람수 등 다른 필드는 정상.
- 미인증(server 모드 세션 만료): 기존 전역 on401 처리 경로를 탄다(다른 API와 동일).
- 빈 `pinEmps` 제출: 정상 — `pinEmps` 미전송 = 전 직원 대상(기존 동작).
