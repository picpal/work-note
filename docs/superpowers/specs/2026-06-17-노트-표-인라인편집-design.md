# 노트 표 인라인 편집기 설계 (Confluence식)

> 작성일 2026-06-17 · work-note frontend · CodeMirror 6 라이브프리뷰 에디터

## 1. 목표

마크다운 표(`| ... |`)를 **실제 표 그리드로 렌더하고 그 위에서 직접 셀을 편집**한다(Confluence식). 파이프를 세는 원문 편집의 고통을 없앤다.

- **단일 진실 = 마크다운 텍스트.** 표 위젯은 뷰 레이어일 뿐, 문서는 항상 GFM 표 마크다운으로 유지된다.
- **내보내기 무손실·무변경.** `note.content`가 그대로 GFM이므로 `.md`/클립보드/PDF 경로(`exportCommands.ts`)는 손대지 않는다.
- **병합 미지원.** GFM은 직사각형 그리드만 표현 → 이 제약이 표↔마크다운 무손실 라운드트립을 보장한다.

### 성공 기준

1. 커서 밖 표는 편집 가능한 그리드로 보인다(현재의 읽기전용 프리뷰 대체).
2. 셀 클릭 → 그 자리에서 텍스트 편집. blur 시 문서 마크다운이 갱신된다.
3. hover 핸들로 행/열 추가·삭제, 열 핸들 메뉴로 좌/중/우 정렬.
4. 모든 편집 후 문서를 내보내면 유효한 GFM 표가 나온다.
5. 표 편집 중 다른 위젯(코드카드·체크박스·이미지·mermaid)·라이브프리뷰 동작은 회귀 없음.

## 2. 현재 상태 (출발점)

`frontend/src/editor/cm.ts`:

- `Table` syntax 노드를 만나면(`buildDecorations`, ~278행) 선택이 표에 닿지 않을 때 `RenderWidget("table", src, nf)`를 **읽기전용** block 위젯으로 그린다. 클릭하면 캐럿을 원문 위치로 보내 raw 마크다운을 노출한다.
- `RenderWidget`는 `renderMarkdown(src)`로 HTML을 만들 뿐 편집 기능이 없다.
- 위젯 범위는 `atomicRanges`(377행)에 들어가 캐럿이 내부로 못 들어간다.
- `decoField`(368행)는 `docChanged || selection || focusChanged` 시 `buildDecorations`를 재실행한다.

→ 본 작업은 **`Table` 분기를 새 `TableWidget`으로 교체**하고 편집·구조조작을 얹는 것이다. 라이브프리뷰 엔진의 나머지는 건드리지 않는다.

## 3. 핵심 설계 결정

| 결정 | 내용 |
|---|---|
| 단일 진실 | 마크다운 텍스트. 위젯은 파생 뷰 |
| 셀 모델 | **평문(plain-text)** — 셀 내용 = 그 칸의 인라인 마크다운 원문 그대로. 셀 안 `**굵게**`는 텍스트로 보이고 그대로 편집 |
| 조작 범위 | 셀 편집 + 행/열 추가·삭제 + **열 정렬(좌/중/우)** |
| 컨트롤 UI | **행/열 hover 핸들 + 핸들 클릭 메뉴**(Confluence식). 정렬은 열 핸들 메뉴에 위치 |
| 병합 | 미지원 (YAGNI, GFM 비표현) |
| 리치 셀 | 미지원 (YAGNI — 셀별 미니에디터·인라인 직렬화 라운드트립 리스크) |

## 5. 데이터 모델 (`gfmTable.ts`)

```ts
export type Align = "none" | "left" | "center" | "right";

export interface TableModel {
  align: Align[];      // 열 개수만큼. 구분행에서 파싱
  header: string[];    // 헤더 셀 원문(파이프 제외, 트림됨)
  rows: string[][];    // 본문 행들. 각 행은 열 개수에 맞춰 정규화
}
```

불변식: `header.length === align.length`, 모든 `rows[i].length === header.length`. 파서가 부족한 셀은 `""`로 채우고 초과 셀은 버려 직사각형을 보장한다.

## 6. 순수 함수 인터페이스 (`gfmTable.ts`)

```ts
// 표 소스(여러 줄 GFM 표 문자열) → 모델. 표가 아니면 null.
export function parseGfmTable(src: string): TableModel | null;

// 모델 → GFM 표 마크다운(끝 개행 없음). 셀은 ` | ` 패딩, 파이프 이스케이프.
export function serializeGfmTable(m: TableModel): string;

// 구조 조작 — 전부 새 모델 반환(불변), 직사각형 불변식 유지.
export function insertRow(m: TableModel, atBodyIndex: number): TableModel;     // atBodyIndex 위치에 빈 행 삽입(끝=push)
export function deleteRow(m: TableModel, bodyIndex: number): TableModel;        // 본문 행만. 0행이면 헤더만 남김
export function insertColumn(m: TableModel, atIndex: number): TableModel;       // 빈 열 삽입, align="none"
export function deleteColumn(m: TableModel, index: number): TableModel;         // 최소 1열 유지(마지막 열은 삭제 거부 → 동일 모델 반환)
export function setAlign(m: TableModel, index: number, align: Align): TableModel;

// 셀 텍스트 헬퍼
export function escapeCell(text: string): string;   // "a|b" → "a\|b", 개행 → "<br>"
export function unescapeCell(raw: string): string;  // 역변환 — 셀 DOM 표시·편집용
```

### 파싱 규칙

- 줄 단위로 분리. 각 줄에서 **이스케이프되지 않은 `|`**로 셀 분리(`\|`는 리터럴 파이프). 선행/후행 `|`는 제거.
- 2번째 줄 = 구분행: 각 셀이 `:?-+:?` 패턴. `:--`→left, `:-:`→center, `--:`→right, `--`→none.
- 구분행이 유효하지 않으면 `parseGfmTable`은 `null`(표 아님) 반환.
- 헤더만 있고 본문 행이 없어도 유효(`rows: []`).

### 직렬화 규칙

- 셀: `escapeCell(text)`, 양쪽 한 칸 패딩 → `| a | b |`.
- 구분행: none→`---`, left→`:---`, center→`:---:`, right→`---:`.
- 라운드트립 보장: 우리가 직렬화한 출력은 다시 파싱하면 동일 모델(정규화 후).

### 알려진 한계 (스펙 명시)

- **인라인 코드 안의 파이프**(`` `a|b` ``)는 셀 분리를 오인할 수 있다. GFM 표준도 셀 내 파이프 이스케이프(`\|`)를 요구하므로, 파서는 이스케이프된 파이프만 리터럴로 취급한다. 직렬화는 항상 이스케이프하므로 우리가 만든 표는 안전.

## 7. `TableWidget` (`tableWidget.ts`)

`RenderWidget`을 대체하는 편집 가능 block 위젯.

```ts
export class TableWidget extends WidgetType {
  private dom?: HTMLElement;     // toDOM에서 보관 (eq·커밋용)
  private view?: EditorView;     // toDOM(view)에서 보관 (blur·디바운스·destroy 커밋용)
  constructor(
    private source: string,   // 표 소스(여러 줄)
    private from: number,      // 문서 내 표 시작 offset
    private to: number,        // 표 끝 offset
  ) { super(); }
}
```

기존 `CheckboxWidget`/`RenderWidget`과 동일하게 **`view`는 `toDOM(view)`로 주입**된다(StateField가 만든 위젯도 toDOM에서 view를 받아 `view.dispatch` 가능 — 현 `CheckboxWidget`이 그 증거). 따라서 생성자·`cm.ts`의 `decoField`(StateField)는 view를 알 필요가 없다. 타이머/destroy 등 toDOM 밖에서 dispatch가 필요한 핸들러는 `this.view`(toDOM에서 보관)를 쓴다.

`serializeFromDom(dom)`은 위젯 내부 헬퍼: DOM 셀들을 읽어 `TableModel`을 만든 뒤(`unescapeCell` 역변환 포함) `serializeGfmTable`로 직렬화한다.

### 7.1 DOM 구조

```
div.cm-table-widget            (contenteditable=false, 래퍼)
  div.cm-table-scroll
    table.cm-table
      tr (헤더)  → th.cm-cell[contenteditable=plaintext-only] ...
      tr (본문)  → td.cm-cell[contenteditable=plaintext-only] ...
  // hover 핸들/삽입버튼 (CSS position, hover 시 표시)
  div.cm-row-handle (행별), div.cm-col-handle (열별)
  button.cm-ins-col, button.cm-ins-row (삽입 ⊕)
  div.cm-table-menu (핸들 클릭 시 표시되는 메뉴, 평소 숨김)
```

- 셀은 `contenteditable="plaintext-only"`로 리치 붙여넣기·서식을 원천 차단(평문 셀 정책과 일치). 셀 표시값 = `unescapeCell(원문)`.
- 래퍼는 `contenteditable=false`라 핸들/메뉴는 편집 대상이 아니다.

### 7.2 toDOM / 포커스 보존 메커니즘 (핵심)

`decoField`는 매 docChange마다 `buildDecorations`를 재실행해 **새 TableWidget을 만든다**. 편집 중 위젯 DOM이 재생성되면 셀 포커스·캐럿이 날아간다. 이를 다음으로 차단한다:

```ts
toDOM(view) { this.dom = build(...); return this.dom; }   // dom 참조 보관
eq(other: TableWidget) {
  // 내 현재 DOM을 직렬화한 결과가 새 위젯의 source와 같으면 → DOM 유지(재생성 안 함)
  return this.dom ? serializeFromDom(this.dom) === other.source : this.source === other.source;
}
```

- 커밋 흐름: 셀 편집은 **DOM 로컬**(문서 미반영). 커밋 시 현재 DOM을 직렬화→`view.dispatch`로 `[from,to]` 치환. 이 dispatch가 유발한 재빌드에서 `other.source == serializeFromDom(this.dom)` → `eq()=true` → **CM이 기존 DOM 유지, 포커스 보존**.
- 외부 변경(undo·프로그램 수정)은 `other.source != serializeFromDom(dom)` → `eq()=false` → 재생성으로 반영. 정확.

### 7.3 커밋 시점

| 시점 | 동작 |
|---|---|
| 셀 타이핑 | DOM 로컬만. 문서 미반영(포커스 안정) |
| 셀 blur(표 밖으로 포커스 이탈) | `serializeFromDom`→dispatch 커밋 |
| 구조 조작(행/열 추가·삭제·정렬) | DOM 즉시 변형 + dispatch 커밋. 직후 포커스 복원 |
| 디바운스(마지막 키 입력 후 1초) | 안전망 커밋(`eq` 덕에 포커스 무손실) |
| 위젯 destroy(노트 전환·언마운트) | `destroy(dom)`에서 미커밋분 최종 커밋(데이터 유실 방지) |

`note.content` 자동저장(60초)·내보내기는 문서 텍스트만 보므로, 위 커밋이 문서를 충분히 신선하게 유지한다.

### 7.4 CM 경계 처리

- `ignoreEvent(e)`: 셀 내부 이벤트는 **true** 반환 → CM이 셀 키 입력을 문서 편집으로 오인하지 않음. Tab/Enter/Escape/메뉴는 위젯이 자체 리스너로 처리.
- `mousedown` 가드: 셀 클릭이 CM 선택을 표 범위로 이동시키지 않도록 처리(셀 포커스는 contenteditable 브라우저 포커스로, CM 선택은 표 밖에 유지). → `selTouches`가 편집 중 false로 유지되어 위젯이 raw로 떨어지지 않음.
- `atomicRanges` 유지: 표 범위는 원자적이라 CM 캐럿이 못 들어간다. 셀 편집은 contenteditable 포커스로 독립 동작.

### 7.5 키보드/상호작용

| 입력 | 동작 |
|---|---|
| 셀 클릭 | 그 셀 편집 포커스 |
| Tab / ⇧Tab | 다음 / 이전 셀. 마지막 셀에서 Tab → 새 본문 행 추가 후 첫 셀 |
| Enter | 아래 셀로 이동(마지막 행이면 새 행) |
| ⇧Enter | 셀 내 줄바꿈(직렬화 시 `<br>`) |
| 행 핸들(⠿, 행 왼쪽 hover) 클릭 | 메뉴: 위 삽입 · 아래 삽입 · 행 삭제 |
| 열 핸들(⠿, 열 위 hover) 클릭 | 메뉴: 왼쪽 삽입 · 오른쪽 삽입 · 열 삭제 · 정렬 ⬅⬛➡ |
| 행/열 사이 ⊕ | 그 위치에 행/열 삽입 |
| 표 전체 삭제 | 표 위/아래 블록 경계에서 선택→삭제(원자 범위 단위). 별도 UI 없음 |

### 7.6 raw 소스 안전판

전용 토글은 두지 않는다. 다만 CM 선택이 표 범위에 걸치면(전체 선택·드래그 선택 등) 기존 `selTouches` 분기대로 위젯 대신 raw 마크다운을 보여준다 — 비용 0의 안전판이며, 평소 셀 편집(브라우저 포커스)은 CM 선택을 건드리지 않아 트리거되지 않는다.

## 8. `cm.ts` 변경

`buildDecorations`의 `Table` 분기(현 278–286행)만 교체:

```ts
if (name === "Table") {
  if (!selTouches(nf, nt)) {
    const firstLine = state.doc.lineAt(nf), lastLine = state.doc.lineAt(Math.min(nt, state.doc.length));
    const src = state.sliceDoc(firstLine.from, lastLine.to);
    out.push(Decoration.replace({ widget: new TableWidget(src, firstLine.from, lastLine.to), block: true })
      .range(firstLine.from, lastLine.to));
    return false;
  }
  return; // 선택이 표에 걸침 → raw
}
```

- `decoField`(StateField)는 그대로 둔다 — 위젯은 `toDOM(view)`로 view를 받으므로 view를 주입할 필요가 없다(§7.1 참조). 변경은 `Table` 분기에서 `RenderWidget("table", ...)` → `new TableWidget(...)` 한 줄뿐.
- `RenderWidget`의 `"table"` 케이스는 제거(더 이상 호출 안 됨). mermaid·codecard 케이스는 유지.

## 9. 테스트 전략

**단위 테스트(`gfmTable.test.ts`, Vitest node)** — 순수 로직 전부 커버:

- `parseGfmTable`: 표준 표, 정렬 마커 4종, 헤더만, 선/후행 파이프 유무, 이스케이프된 `\|`, 불규칙 열 수 정규화, 표 아님→null.
- `serializeGfmTable`: 정렬 마커 출력, 파이프 이스케이프, 패딩.
- **라운드트립 속성**: `parse(serialize(m))`가 정규화된 `m`과 동일. 여러 모델 픽스처로.
- `insertRow/deleteRow/insertColumn/deleteColumn/setAlign`: 직사각형 불변식 유지, 경계(마지막 열 삭제 거부, 본문 0행에서 행 삭제).
- `escapeCell/unescapeCell` 왕복.

**위젯/CM 통합** — 프로젝트 관례상 컴포넌트 렌더 테스트는 두지 않는다(@testing-library 미도입). 대신 스펙에 **수동 QA 체크리스트**를 둔다:

1. 셀 편집 후 blur → 문서 마크다운 갱신 확인
2. Tab 끝 셀 → 새 행, ⇧Enter → `<br>`
3. 행/열 추가·삭제, 좌/중/우 정렬 → 내보내기 GFM 유효
4. 편집 중 위젯 포커스 유지(중간 재렌더에도 캐럿 안 날아감)
5. undo/redo로 표 변경 정확 반영
6. 코드블록·체크박스·이미지·mermaid 위젯 회귀 없음
7. `\|` 포함 셀 왕복, 빈 셀, 헤더만 있는 표

## 10. 내보내기 영향

없음. `buildMarkdown`(`exportCommands.ts:15`)은 `note.content`를 그대로 사용. 표 편집이 문서 텍스트를 GFM으로 유지하므로 `.md`·클립보드·PDF 전부 무변경.
